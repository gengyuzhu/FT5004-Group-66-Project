# MilestoneVault Architecture

## System split

MilestoneVault keeps only trust-minimized state and ETH custody on-chain:

- campaign ownership, goal, deadlines, total raised, milestone status
- milestone proof CIDs
- weighted vote tallies
- creator withdrawable balance
- refund eligibility and refund claims

Everything content-heavy stays off-chain:

- campaign title, summary, description, milestone prose, cover image
- milestone evidence files
- proof summaries and demo links

## Contract model

The `MilestoneVault` contract supports multiple campaigns and enforces:

1. fundraising escrow
2. public finalization after deadline
3. sequential milestone proof submission
4. contribution-weighted voting
5. phased creator withdrawals after approved milestones
6. refund-only access to unreleased escrow after failure
7. public failure for missed milestone due dates

### Status flow

- `Fundraising`
- `Active`
- `Completed`
- `Failed`

### Failure flow

- underfunded fundraising
- rejected milestone vote or missed quorum
- missed milestone deadline before proof submission

## Frontend model

The Next.js frontend uses:

- `wagmi` + `viem` for wallet and contract reads/writes
- a compact topbar plus dashboard/create/detail surfaces tuned for desktop and mobile layouts
- direct search/filter/pagination in the campaign directory before introducing a heavier indexing layer
- direct contract reads plus event logs instead of The Graph
- server-side Pinata API routes to pin files/JSON without exposing the JWT to the browser

The frontend reads `web/src/lib/contracts/milestoneVault.ts`, which is generated by the Hardhat export script after compile/deploy.

The repository `docs/` site mirrors the same product language in a static format so GitHub Pages can showcase the interaction model without requiring a live backend.

For local browser E2E, the frontend also exposes a mock IPFS route. This lets Playwright exercise the upload and retrieval flow end to end without depending on live Pinata during automated runs.

## IPFS flow

### Campaign creation

1. User enters campaign metadata and optionally uploads a cover image.
2. The frontend POSTs a multipart form to `/api/ipfs/campaign`.
3. The route uploads the cover image to Pinata, builds a metadata JSON bundle, and pins that JSON.
4. The returned CID is written on-chain via `createCampaign`.

### Milestone proof submission

1. User uploads files and proof summary.
2. The frontend POSTs to `/api/ipfs/proof`.
3. The route uploads each file, builds a proof JSON bundle, and pins it.
4. The returned CID is written on-chain via `submitMilestoneProof`.

## Security posture

- OpenZeppelin `ReentrancyGuard`
- withdraw pattern for creator payouts and backer refunds
- no upgradeability/proxy layer in MVP
- no iteration over backers on-chain
- no PII stored on-chain
